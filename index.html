🏥 ClaimHeart — Complete Project Breakdown

This document explains every file, folder, pipeline, and workflow in the ClaimHeart codebase. Written in plain language so you can understand exactly what is happening at every level of the system.

What is ClaimHeart?
ClaimHeart is an AI-powered medical insurance claims processing platform. It reads scanned hospital documents, checks them against insurance policies, detects fraud, and writes decision letters to patients — all automatically. Three dashboards serve three users: the Insurance Company, the Hospital/TPA, and the Patient.

3. backend/app/main.py — FastAPI Entry Point

backend/app/main.py
This is the FIRST file that runs when you start the server. It creates the FastAPI application, adds CORS middleware (so the frontend can call the backend from a different port), and registers all the URL routes.

When you run uvicorn app.main:app --reload, Python reads this file. Here is what it does step by step:

1. Creates a FastAPI() app with a title, description, and version.
2. Adds CORSMiddleware — this allows http://localhost:3000 (the Next.js frontend) to call http://localhost:8000 (the backend) without browser security errors.
3. Imports routers from api/routes/ and registers them with URL prefixes:
  • health.router → handles /api/health
  • ocr.router → handles /api/ocr/upload and /api/ocr/process-local
  • claims.router → handles /api/claims/...
4. Adds a root route GET / that returns a simple JSON confirming the server is running.
Why this matters: Every API call from the frontend or from testing goes through routes registered in main.py. If a route is not registered here, it doesn't exist. This file is the traffic controller of the entire backend.

4. backend/app/agents/ — The 4 AI Agents

This folder contains the 4 core AI agents that process every claim. Each agent is a subfolder with its own logic. They are meant to run in parallel, though right now only the Extractor Agent has real code written.

4.1 Extractor Agent — agents/extractor/

This agent reads uploaded medical documents (PDFs, images) and turns them into structured data (JSON).

agents/extractor/agent.py
The main function here is extractor_agent(file_path). It does two things in sequence: (1) calls extract_text_from_image() from utils/ocr.py to get raw text from the image using EasyOCR, then (2) calls parse_medical_text() from utils/parser.py to find patient name, age, medications, diagnosis etc. from that raw text. Returns a dict with both raw_text and structured_data.

agents/extractor/output_schema.py

agents/extractor/pdf_parser.py

agents/extractor/prompts.py

6. backend/app/services/ — Business Logic Layer

Services sit between the API routes and the agents/database. They coordinate multiple operations together.

services/extractor_agent.py
This is the most complete service file. Contains the ExtractorAgent class that uses Google Gemini AI to extract structured data from medical documents.

How it works:
  1. Takes raw OCR text as input
  2. Calls _identify_document_type() — sends the first 1000 characters to Gemini and asks it to classify whether this is a "bill", "diagnostic_report", "discharge_summary", "pharmacy", or "claim"
  3. Based on the document type, selects the matching schema from schemas/ folder (e.g., BILL_PROMPT for bills, DISCHARGE_SCHEMA for discharge summaries)
  4. Sends the full OCR text + the schema prompt to Gemini with response_mime_type="application/json" to force JSON output
  5. Returns the extracted structured JSON
Uses the google-generativeai library with gemini-2.0-flash model. API key comes from GEMINI_API_KEY environment variable.
services/llm_parser.py
An earlier/alternative extractor using OpenAI GPT-4o-mini instead of Gemini. Sends raw text to ChatGPT and asks for a JSON with hospital_name, total_charges, insurance_portion, patient_portion, and billing items. Note: This file prints the API key in the console (debug artifact — should be removed before production).
services/claim_service.py
Currently empty — planned to coordinate the full claim lifecycle: receiving a new claim, triggering extraction, triggering policy check, fraud check, and letter generation.
services/fraud_service.py, policy_service.py, letter_service.py
Currently empty — planned to be service classes for each agent domain, coordinating database reads/writes and agent calls.

7. backend/app/tasks/ — Async Celery Workers

Tasks are functions that run in the background asynchronously. When a hospital uploads a 20-page PDF, the HTTP request shouldn't wait 30 seconds — instead it queues the work as a task and returns immediately. Celery (backed by Redis) handles this queuing.

tasks/extraction.py
The only task with working code. Contains process_document(file_path) which is a thin wrapper — it just calls extractor_agent(file_path) and returns the result. This is the entry point that api/routes/ocr.py calls.

The call chain: OCR route → process_document() → extractor_agent() → extract_text_from_image() + parse_medical_text()
tasks/analysis.py
Currently empty — planned to hold the async task that runs the Policy Agent and Fraud Investigator Agent after extraction is complete.
tasks/communication.py
Currently empty — planned to hold the async task that runs the Mediator Agent and sends letters/notifications after analysis is complete.
tasks/worker.py
Currently empty — planned to initialize the Celery application with Redis as the message broker. Would look something like: celery_app = Celery("claimheart", broker="redis://localhost:6379/0")

8. backend/app/utils/ — Utility Functions

Pure helper functions used by agents and services. No business logic — just tools.

utils/ocr.py
Most important utility file. Contains extract_text_from_image(image_path).

Uses EasyOCR (a Python library) to read text from images. EasyOCR supports handwritten and low-quality scanned documents. The function:
  1. Creates an EasyOCR Reader for English language at module load time (global variable — only done once)
  2. Calls reader.readtext(image_path, detail=0) which returns a list of text strings found in the image
  3. Joins them all with newlines and returns a single string
Note: The file also has a commented-out version using Tesseract OCR (a different OCR library). EasyOCR was chosen because it handles messy scanned documents better.
utils/image_processing.py
Contains preprocess_for_ocr(image_path) which improves image quality before OCR:
  1. Reads the image in grayscale using OpenCV
  2. Scales it up 2x (so small text becomes bigger and easier to read)
  3. Applies Adaptive Thresholding — converts grey pixels to pure black or white. This removes shadows and uneven lighting common in scanned documents
Returns a PIL Image object ready for Tesseract OCR. Used by core/ocr_engine.py.
utils/parser.py
The largest utility file. Contains parse_medical_text(raw_text) which calls 7 sub-functions to extract specific fields using regex patterns:
  • extract_patient_name() — looks for patterns like "Patient Name: John Doe" or names before "(ICD"
  • extract_age() — looks for "age: 32" or "32 years"
  • extract_gender() — looks for "sex: male/female"
  • extract_medical_record_number() — looks for patterns like "CCH-OPD-2026-09341"
  • extract_hospital_info() — looks for hospital/clinic keywords and phone numbers
  • extract_diagnosis() — looks for "diagnosis:", "condition:", "patient with" patterns
  • extract_medications() — looks for "Tab", "Inj", "Syp" prefixes + known drug names like Paracetamol, Pantoprazole
  • extract_vital_signs() — looks for "BP:", "pulse:", "temperature:" patterns
  • extract_dates() — looks for admission date, discharge date, prescription date
Returns a dictionary with all these fields. Many patterns handle OCR errors (e.g., "turnule" instead of "female", "rutu" instead of "pulse").
utils/cms_baseline.py, utils/cost_baselines.py
Currently empty — planned to hold regional cost reference data (e.g., average cost of appendectomy in Mumbai = ₹50,000). The Fraud Investigator Agent would use this to detect when a hospital charges 5x the normal rate.
utils/redaction.py
Currently empty — planned to use Microsoft Presidio to strip PII (patient name, phone, ID numbers) from documents before sending them to external AI APIs. Required for HIPAA/DPDP compliance.
utils/s3_helpers.py, utils/textract.py, utils/pdf_helpers.py
All currently empty. Planned for: AWS S3 file storage, AWS Textract OCR as fallback, and PDF-specific parsing helpers (page splitting, image conversion).

9. backend/app/schemas/ — Data Structures

Schemas define what fields different types of medical documents contain. They are used as prompts for AI models — they tell the AI what to look for.

schemas/bill_schema.py
Contains BILL_PROMPT — a text prompt that tells Gemini to extract: hospital details (name, address, registration), patient details (name, age, gender, ID), billing items (description, quantity, unit price, tax, total), totals (subtotal, tax, discount, grand total), and dates (bill date, admission, discharge).
schemas/claim_schema.py
Contains CLAIM_SCHEMA — a dictionary template showing the shape of a claim record: claim_id, policy_number, claim_type (Pre-Authorization or Reimbursement), hospital network status, estimated cost, diagnosis, and whether it's an emergency.
schemas/discharge_schema.py
Contains DISCHARGE_SCHEMA — template for discharge summaries: admission date, discharge date, reason for admission, treatment given, condition at discharge, follow-up advice, and ICD-10 codes.
schemas/pharmacy_schema.py
Contains PHARMACY_PROMPT — prompt for pharmacy bills: pharmacy name, GST number, bill number, list of medicines (name, batch, expiry, quantity, price, tax), and prescribing doctor.
schemas/report_schema.py
Contains REPORT_SCHEMA — template for lab test reports: test name, test date, observed values (parameter, result, unit, reference range), impression (doctor's conclusion), and pathologist name.
schemas/fraud.py, schemas/letter.py, schemas/policy.py, schemas/token.py, schemas/user.py
All currently empty — planned Pydantic schemas for request/response validation on the API endpoints.

11. backend/app/ml/ — Machine Learning

The ML folder holds the statistical fraud detection model and related utilities.

ml/isolation_forest.py
Planned: Defines and trains a scikit-learn Isolation Forest model. Isolation Forest is an unsupervised anomaly detection algorithm — it finds data points that are "isolated" from the rest. In the fraud context: if a hospital bills ₹5 lakh for dengue when the average is ₹1 lakh, the ₹5 lakh claim gets a high anomaly score.
ml/data_prep.py
Planned: Converts claim JSON data into numerical feature vectors that the ML model can process. For example: converts "3 days hospitalization + ₹50,000 cost + 5 tests" into a numerical array [3, 50000, 5].
ml/model_registry.py
Planned: Saves and loads trained ML model weights. So you train the model once on historical data and then load it for each new claim.
ml/evaluate.py
Planned: Measures how accurate the fraud model is using historical labeled data (known fraud cases).
ml/test_ocr.py
A standalone test script — not really ML. Processes a specific image file at temp_uploads/sample_image2.jpg, prints the OCR output, and saves it to ocr_output.json. Used during development to quickly test OCR quality.

12. backend/app/pipelines/ — RAG Pipeline

RAG stands for Retrieval Augmented Generation. Instead of asking an AI "does this claim cover dengue?", RAG first searches the actual policy PDF for relevant sections, then sends those sections + the question to the AI. This gives much more accurate and citable answers.

pipelines/rag_orchestrator.py
Planned: Coordinates the full RAG pipeline using LlamaIndex. Takes a claim and a policy, retrieves the most relevant policy sections using vector similarity, and generates a coverage decision with exact citations.
pipelines/document_chunking.py
Planned: Splits long policy PDFs into smaller overlapping text chunks (e.g., 500 words each with 50-word overlap). Smaller chunks are easier for the AI to search through meaningfully.
pipelines/query_engine.py
Planned: The search component — takes a natural language query ("is dengue covered after 6 months?"), converts it to a vector embedding, searches Pinecone for the closest matching policy chunks, and returns them.

13. backend/app/core/ — Config, Logging, Security

core/ocr_engine.py
Contains the ClaimHeartOCR class — a wrapper around Tesseract OCR (different from the EasyOCR in utils/ocr.py).
  • extract_from_image(image_path) — preprocesses the image using image_processing.preprocess_for_ocr() then runs Tesseract with OCR Engine Mode 1 (neural network) and Page Segmentation Mode 3 (fully automatic)
  • extract_from_pdf(pdf_path) — converts each PDF page to an image using pdf2image, then runs Tesseract on each page and concatenates the text
This is used by tests/agents/test_extractor.py to test the full pipeline.
core/config.py
Planned: Loads environment variables (database URL, API keys, Redis URL, JWT secret) using Pydantic BaseSettings. Makes all config available as typed Python objects throughout the app.
core/logging.py
Planned: Configures structured JSON logging with trace IDs so you can track a single request across all the agent calls.
core/security.py
Planned: Password hashing (bcrypt), JWT token creation and verification for user authentication.
core/exceptions.py
Planned: Custom exception classes and global error handlers that return standardized JSON error responses.

14. Backend Standalone Scripts

These Python files in the backend/ root are standalone scripts for testing and batch operations. You run them directly with python filename.py.

backend/extract_image.py
Processes a hardcoded file app/docs/image.png. Prints the extracted text and structured data to console, then saves to ocr_output.json. Quickest way to test OCR on a sample file.
backend/extract_ocr.py
Flexible version — accepts command-line arguments: python extract_ocr.py image.jpg -o results/output.json. Shows the first 500 characters of extracted text in the console, full data in the output file. Has a hardcoded default path for Windows (D:\claimHeart\backend\temp_uploads\image.png).
backend/batch_extract.py
Processes ALL images in a folder. Usage: python batch_extract.py temp_uploads/. Saves each image's JSON result as imagename_ocr.json in the outputs folder. Also creates a summary JSON with success/failure counts. Good for processing many claims at once.
backend/process_image.py
Similar to extract_image.py but uses the image filename for the output (e.g., image.png → outputs/image.json). Shows structured data in console.
backend/test_api.py
Tests the running API server. Calls /api/health and /api/ocr/process-local and prints the responses. You must start the server first with uvicorn, then run this script.

lib/api/auth.ts
Handles user authentication — all stored in localStorage (demo mode). Key functions:
  • getCurrentUser() — reads the current user from localStorage key "claimheart.currentUser"
  • loginUser(email, password, role) — finds a matching user in the "users" localStorage array, sets them as current user
  • signupUser(payload) — creates a new user object, adds it to the "users" array in localStorage, sets as current
  • logout() — clears localStorage keys and redirects to /login

lib/api/claims.ts
All claim operations. Key functions:
  • submitClaim(claimInput) — creates a new Claim object with generated ID, adds it to Zustand store (which saves to localStorage), and fires notifyClaimSubmitted()
  • recordDecision(id, status, note) — updates a claim's status, appends a timeline entry, and fires notifyDecisionMade()
  • sendDecisionEmail(id) — creates a ClaimEmail object, appends it to the claim's emails array, fires notifyDecisionEmailSent()

lib/api/notifications.ts
Creates and adds notifications to the Zustand store. Each function targets the right roles:
  • notifyClaimSubmitted(claim) → sends to "insurer" (new claim alert) and "patient" (your claim was submitted)
  • notifyDecisionMade(claim, decision, note) → sends to "patient" (approved/denied message) and "hospital" (decision notification)
  • notifyDocumentRequested(claim, note) → sends to both "patient" and "hospital" (action required)
  • notifyDocumentUploaded(claim, fileName, role) → sends to "insurer" (new document ready to review)
  • notifyDecisionEmailSent(claim, subject) → sends to "patient" (email was sent) and "hospital" (patient was notified)
lib/api/storage.ts
Two simple functions: readStorage(key, fallback) reads and JSON-parses a localStorage key (returns fallback if not found or invalid JSON), and writeStorage(key, value) JSON-stringifies and saves to localStorage. Handles SSR by checking for window existence. Used by auth.ts and throughout the app.

19. frontend/store/ — Zustand State Management

store/useAppStore.ts
The main global state store using Zustand. Holds ALL claims and notifications in memory. Key features:
  • claims: Claim[] — array of all claims, deduplicated by ID
  • notifications: Notification[] — array of all notifications
  • lastSyncAt: number — timestamp of last state change (watched by LiveBadge)
  • addClaim(claim) — prepends new claim, saves to localStorage
  • updateClaim(id, updates) — merges updates into existing claim, saves to localStorage
  • addNotification(notification) — prepends new notification, saves to localStorage
  • markNotificationRead(id) — marks one notification as read, avoids unnecessary re-renders if already read
  • markAllNotificationsRead(role, userId) — marks all matching notifications as read (matches by role and optionally by userId)
  • syncFromStorage() — reads both "claims" and "notifications" from localStorage and updates the in-memory state. Called on page load and on the "storage" window event (so multiple tabs stay in sync)
The deduplication function (dedupeById) prevents the same claim appearing twice — important because claims can come from both seeding and live submission.
store/useClaimStore.ts
A lighter-weight, custom store for claim selection state. Uses React's useSyncExternalStore directly (no Zustand). Holds: the full claims array, the currently selected claim, and a hydrated flag. Has actions: setClaims, selectClaim, updateClaim. Used in some components that need fine-grained claim selection tracking without the full Zustand overhead.

20. frontend/hooks/ — Custom React Hooks

hooks/usePageReady.ts
Returns a boolean ready that starts as false and becomes true after the first browser animation frame (or after an optional delay).

Why this is needed: Next.js renders pages on both the server (where localStorage doesn't exist) and the client. If you show localStorage data during server render, you get a hydration mismatch error. By returning false initially, components show skeleton loaders during SSR, then show real data only on the client.
hooks/useSyncStore.ts
Called inside DashboardLayout to initialize the app state. On mount: calls seedIfEmpty() to ensure demo data exists, then calls syncFromStorage() to load it into Zustand. Also adds a "storage" event listener so that when localStorage changes in another browser tab (e.g., you submit a claim in the hospital dashboard), this tab automatically re-syncs and shows the new claim.
hooks/useClaims.ts, useFraudScore.ts, useDebounce.ts
All currently empty. Planned: useClaims would use React Query to fetch claims from the backend API with caching. useFraudScore would poll the backend for fraud score updates. useDebounce would delay search input to avoid excessive filtering on every keystroke.

22. Complete Data Flow — End to End

Backend OCR Data Flow

1. Someone calls POST /api/ocr/upload with a file or POST /api/ocr/process-local with a path.
2. api/routes/ocr.py validates the file (type, size), saves temporarily if uploaded.
3. Calls tasks/extraction.py → process_document(file_path)
4. Which calls agents/extractor/agent.py → extractor_agent(file_path)
5. Which calls utils/ocr.py → extract_text_from_image(file_path)
6. EasyOCR reads the image, returns a list of text strings, joined into one string.
7. Then calls utils/parser.py → parse_medical_text(raw_text)
8. 8 regex functions extract patient name, age, gender, medications, dates, etc.
9. Returns {"raw_text": "...", "structured_data": {...}} back up the chain to the API response.

Frontend Demo Workflow Data Flow

1. User visits the app. AppBootstrap → seedIfEmpty() → writes demo data to localStorage if not already there.
2. DashboardLayout mounts → useSyncStore() → syncFromStorage() → Zustand loads all claims and notifications from localStorage into memory.
3. Hospital Dashboard: User selects Case 2 (Dengue Fraud). demoWorkflow.loadDemoDocumentCorpus() fetches 5 text files from public/docs/.
4. Hospital uploads files to the 3 required slots (prescription, lab report, billing). Each slot is stored as a DemoUploadSelection in component state.
5. "Run Intake Scan" → for each document slot: shows scanning animation → shows extraction animation → types out the document text character by character.
6. "Submit Claim" → createWorkflowClaimInput() builds a Claim object → lib/api/claims.submitClaim() → store.addClaim() → saves to localStorage.
7. notifyClaimSubmitted() → creates 2 notifications (one for insurer, one for patient) → store.addNotification() twice → saves to localStorage.
8. If another tab is open (insurer dashboard), the "storage" window event fires → syncFromStorage() → the new claim appears in the insurer's queue.
9. Insurer clicks "Start AI Scan" on their dashboard. The scanning stages run with setTimeout delays. Each stage updates component state (not Zustand — this is local animation state).
10. After all stages complete: store.updateClaim(id, finalDecision) → claim status, decision letter, amountApproved, and pipelineCompletedAt are updated in Zustand and localStorage.
11. notifyDecisionMade() → 2 more notifications → patient and hospital are notified.
12. Patient dashboard re-renders with the updated claim status and decision letter (read from Zustand store which reads from localStorage).

23. The Full Agent Pipeline — Step by Step

This is how a real claim SHOULD flow through the complete system (partly built, partly planned):

Step 1: Document Intake
Hospital staff scan pre-auth form, discharge summary, lab reports, billing invoice → Upload via hospital dashboard → Files hit POST /api/ocr/upload
↓
Step 2: Extractor Agent (BUILT)
process_document() → EasyOCR reads every page → parse_medical_text() extracts patient name, diagnosis, medications, amounts → Returns structured JSON. Then ExtractorAgent (Gemini) classifies document type and extracts using the right schema.
↓
Step 3: PII Redaction (PLANNED)
Microsoft Presidio strips patient name, phone, ID from document copies before sending to AI models → Creates redacted copies stored separately in S3.
↓
Step 4: Policy RAG Agent (PLANNED)
LlamaIndex retrieves matching policy clauses from Pinecone vector store → LLM reasons over them → Returns: covered/partially_covered/denied + exact clause citations with page numbers + amounts.
↓
Step 5: Fraud Investigator Agent (PLANNED — simulated in frontend)
Rule engine checks: duplicate claims, waiting period, test count vs protocol → Isolation Forest checks cost anomaly vs regional baseline → LLM reasons over findings → Returns: fraud_score 0-100, evidence list, recommendation (approve/query/reject/dispatch_agent).
↓
Step 6: TAT Monitor (PLANNED — simulated)
Checks SLA timers — was initial approval given within 1 hour? Is discharge approval due within 3 hours? If breach risk, escalates to senior reviewer and notifies hospital.
↓
Step 7: Decision Aggregator (PLANNED — simulated)
Merges all 3 agent outputs → Resolves conflicts → Produces final decision packet: status + approved_amount + rationale + evidence.
↓
Step 8: Mediator Agent (PLANNED — built in frontend)
Generates 3 communications: (1) Patient letter in simple language — approved X, denied Y, here's why, here's what to do next. (2) Hospital query (if docs missing). (3) Insurer summary with full audit trail.
↓
Step 9: Notifications and Dashboards
All 3 dashboards update. Patient sees decision letter. Hospital sees query or approval. Insurer sees explainable decision with evidence. SMS/email sent to patient.
